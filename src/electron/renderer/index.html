What this actually is

    

      Docker can run almost any app — websites, databases, APIs — each in its own
      sealed box, so nothing an app does can tangle with the rest of this computer. But Docker on its own
      does exactly what it's told, exactly once: if the app crashes at 2 a.m., it stays crashed.
      Single-machine orchestration on the plain Docker Engine API. An app is a
      kh/v1 manifest — built by this GUI's wizard or written by hand; both pass the
      same zod schema and the same reconciler.
    

    

      kube-helper is the layer that takes responsibility. You describe what should be
      running; it makes that true, then keeps it true — through crashes, updates and scale changes. The ideas
      are borrowed from Kubernetes, the system datacentres use, minus everything that needs a team to babysit.
      There is no control plane, no etcd, no database of any kind: every byte of kh
      state rides on container labels, so docker ps never disagrees with
      kh status, and deleting the containers deletes the state.
    

  


  

    
The loop that does all the work

    

      Everything this app does — deploying, scaling, updating, healing — is one trick, repeated:
      Every feature is the same reconcile primitive:
    

    

      

        
What you asked forDesired state

        
my-blog · 2 copiesreplicas
nginx:alpine · port 8080

      

      
▶

      

        
What is actually runningObserved state

        
copyreplica 0 running
copyreplica 1 crashed

      

    

    
↻ 
      kube-helper compares the two and fixes the difference — here, it would bring copy 1 back
      diff desired against observed, converge only the difference — here, re-create replica 1
    

    

      Deploying is "the difference is the whole app". Scaling is "the difference is
      one copy". An update is "the difference is the version" — copies are replaced one at a time, so the app
      never goes down. And a crash is just a difference too, so it gets fixed like one.
      Apply, scale, edit and crash-recovery are all this diff-and-converge: per-index,
      so untouched replicas never restart; image changes roll one index at a time;
      restart: always plus kh watch close the loop between runs.

      Everything this window does, a terminal command called kh
      does too — same engine underneath, identical results. You never have to touch it; but if a developer
      hands you commands, they'll work on the very apps you made here.
      The GUI calls the same functions the CLI binds — form → manifest → schema →

      via kh apply -f app.yaml.

When something misbehaves

    
"Docker: not running" in the header — Docker Desktop isn't started. Open it, wait for
    "Engine running", press Retry. kube-helper can't run anything without it.

    
"port is already allocated" — another program (or another app here) owns that
    localhost port. Pick a nearby one — 8081, 8090, 3000…

    
Deploying hangs the first time — it's downloading the app package. Only the first
    deploy of each package pays this; after that it's seconds.

    
An app keeps restarting — open Logs ⧉ and read the red lines: that's the app
    itself telling you what's missing. Databases usually want a password setting
    (e.g. POSTGRES_PASSWORD).

    
A copy shows "unhealthy" — it's running but not answering its health check: wedged
    or overloaded. The terminal's kh watch re-creates such copies automatically.

    
And when in doubt: nothing here is precious. Delete the app and
    deploy it again — it takes seconds, and kept data survives.
